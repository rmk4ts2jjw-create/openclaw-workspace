# Decisions Made

### Decision: Framer Motion for Agent Office animations (no game engine)
Reason: Framer Motion integrates directly with React, no additional runtime needed.
Sprite-based animation with CSS transforms is sufficient for the visual style.

### Decision: Event-driven state management via stationReducer
Reason: Centralized state makes it easier to trigger visual changes from anywhere
in the codebase. Reduces prop drilling across 20+ components.

### Decision: Two-layer auto-resolution (real-time timeline + bulk cron)
Reason: Real-time for when tasks complete naturally. Bulk for cleanup of stale
incidents and cron-triggered resolution. Separation avoids side effects.

### Decision: URL search params for filter state (not React state)
Reason: Filter buttons using `onClick` + `useState` depended on React hydration.
Converting to `<Link>` with URL search params works even without JS hydration.
This fixed a long-standing filter bug.

### Decision: OpenCode + OpenRouter for coding delegation
Reason: Local Ollama models on 16GB Mac cannot support tool-calling agents.
OpenCode with qwen3-coder via OpenRouter credits is the only viable coding agent.
Cost: ~$0.27 per complex task. Credits reserved for coding only.

### Decision: Review system phased approach (6 phases)
Reason: Validate the concept with manual review before investing in automation.
Each phase gates the next. Prevents building expensive automation for a process
that doesn't produce value.

# Risks Identified

### Risk: `tasks.json` concurrent write data loss
Both shell scripts and MC API write to `tasks.json`. Shell script lacks atomic
write guard. API uses tmp+rename. Simultaneous writes could cause data loss.
Likelihood: low. Impact: medium.

### Risk: Incident dedup by exact title match
Same root cause with different counts creates separate incidents (e.g., "11 429
errors" vs "26 429 errors"). No fuzzy grouping. Historical evidence: 12+ separate
rate-limit incidents for the same underlying issue.

### Risk: Client-side staleness on Dashboard
Dashboard fetches tasks/incidents at page load. No real-time refresh. Changes
via heartbeat/API not reflected until navigation. Sidebar badge polls every 30s
but main page doesn't.

### Risk: Agent Office animation performance
Framer Motion with 20+ animated components, sprite rendering, and ambient effects
could cause jank on lower-end devices. No performance profiling done yet.

### Risk: Review system produces noise not value
External AI review may not provide actionable recommendations for routine work.
Risk of generating expensive noise if review packages are too detailed or too
frequent. Phase 2 manual testing needed to validate.

# Known Limitations

- No real-time updates on Dashboard (requires navigation to refresh)
- No per-job retry configuration (global 3-retry policy only)
- No exponential backoff on retries
- Incident dedup is title-exact-match only (no fuzzy grouping)
- Agent Office animation performance not profiled
- Review system Phase 1 only (no external AI integration)
- OpenCode credits limited ($10) — sustainable but not unlimited
- Night Shift limited to 2 tasks/night (conservative)
- No browser automation for external review (Phase 2+)

# Questions For Reviewer

1. **Architecture:** Is the two-layer auto-resolution (real-time + bulk cron) the
   right pattern, or should this converge to a single mechanism?

2. **Incident lifecycle:** Should rate-limit incidents be grouped into a single
   rolling incident instead of creating new ones each detection?

3. **Agent Office:** Is the Framer Motion + sprite approach scalable, or will it
   hit performance walls as more agents/rooms are added?

4. **Review system:** Is the 6-phase approach too conservative? Could we skip
   to automated review (Phase 3) given that OpenCode + OpenRouter is already
   working well for coding?

5. **Coding delegation:** Is the OpenCode + qwen3:coder workflow sustainable
   long-term, or should we invest in local model capability (e.g., larger Mac
   or eGPU)?

6. **Data integrity:** Should `tasks.json` be migrated to a proper database
   (SQLite/PostgreSQL) to eliminate concurrent-write risk and enable proper
   querying?
