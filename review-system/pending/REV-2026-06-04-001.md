# Decisions Made

### Decision: Health states integrated into existing Scheduler page (not a new page)
Reason: Avoid page bloat. The Scheduler already shows all cron jobs. Adding health
summary bar and status dots was zero-cost visually and reused existing data
(`consecutiveErrors`, `enabled`).

### Decision: Simple retry policy (3 retries + 5min gap, no exponential backoff)
Reason: Phase 1 — get the basics working. Per-job customization and exponential
backoff deferred to Phase 5. Global policy is sufficient for now.

### Decision: Incident auto-resolve via `taskStatus` field on timeline endpoint
Reason: The timeline endpoint is already the hook point for task→incident sync.
Adding `taskStatus` as an optional field means auto-resolution happens naturally
without a separate call. Falls back gracefully if `taskStatus` is not sent.

### Decision: Bulk auto-resolve endpoint separate from real-time resolve
Reason: Two different use cases. Real-time (timeline endpoint) for when tasks complete.
Bulk (auto-resolve endpoint) for cleanup of stale incidents and cron runs. Keeping
them apart avoids side effects.

### Decision: Linked tasks created even when TRIAGE incident already exists
Reason: Previous dedup logic skipped everything — including task creation — when it
found an existing active incident. This meant repeat detections never got linked tasks,
breaking the entire auto-resolution chain.

### Decision: Backlog count badge uses separate polling endpoint
Reason: `getBacklogCount` is lighter than `getTasks`. The sidebar polls every 30s —
no need to serialize the full task list just for a count.

# Risks Identified

### Risk: `tasks.json` data loss during concurrent writes
Both the auto-detect shell script and the MC API write to `tasks.json`. The shell
script has no atomic write (no `rename` guard). The API (`saveTasks`) does use
`tmpPath + rename`. If both write simultaneously, data loss is possible.
Likelihood: low (30-min cron vs interactive API calls). Impact: medium.

### Risk: Deduplication by title match may merge distinct incidents
The auto-detect script deduplicates by exact title + source tag match. If the same
underlying issue triggers with different counts (e.g., "11 429 errors" vs
"26 429 errors"), they get different titles and are NOT deduplicated. This creates
multiple incidents for the same root cause. The 12+ historical rate-limit incidents
are evidence of this.

### Risk: Client-side task data staleness
The Dashboard fetches tasks at page load via the loader. If a task status changes
via the heartbeat or API, the Dashboard won't reflect it until next navigation.
The sidebar badge polls every 30s but the main page doesn't.

### Risk: `ts` field inconsistency across tasks
Most tasks use `ts: "just now"` (non-parseable), some use ISO timestamps. The
stallend detection logic has to handle both. This is fragile — a future refactor
should enforce ISO timestamps everywhere.

# Known Limitation

- No real-time updates on the Dashboard (requires navigation to refresh task/incident data)
- No per-job retry configuration (global 3-retry policy only)
- No exponential backoff on retries
- Incident dedup is title-exact-match only (no fuzzy grouping)
- Browser automation for external review not implemented (Phase 1 is framework only)
- The `/api/incidents/auto-resolve` endpoint exists but is triggered only by
  cron/heartbeat — no UI button to run it manually
