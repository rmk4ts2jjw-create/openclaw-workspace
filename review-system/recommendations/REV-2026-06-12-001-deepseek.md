## Question-Specific Recommendations

### 1. Two-Layer Auto-Resolution Architecture
**Recommendation:** Converge to a single mechanism.

The real-time + bulk cron split creates subtle consistency issues. If heartbeat resolves between cron runs, the bulk cron will skip it with different resolution criteria.

**Action:** Move bulk cron to continuous loop with lightweight queue (SQLite polling table) OR reduce cron to reconciliation safety net that logs discrepancies without acting.

### 2. Rate-Limit Incident Grouping
**Recommendation:** Use rolling incident with metadata counters.

Creating 12+ separate incidents for same rate-limit condition fragments history.

**Action:** Implement dedup key (error_type, endpoint_pattern, time_window). Upsert into existing open incident within window, increment detection_count.

### 3. Agent Office Framer Motion Scalability
**Recommendation:** Profile first, prepare escape hatches.

20+ animated components need profiling.

**Action:** Profile on lowest-spec device (4x CPU throttle). If <30fps, implement viewport-culling via IntersectionObserver or useReducedMotion toggle.

### 4. Review System: Skip to Phase 3?
**Recommendation:** No. Phase 2 validation is critical but compress it.

The risk is unknown actionability. You'll discover this during manual testing.

**Action:** Run Phase 2 concurrently with Phase 1. Pick 5 real tasks, manually generate reviews, ask "Did this change what I did?" If <60% cause action, tighten templates. Compress to 2 weeks max.

### 5. Coding Delegation Sustainability
**Recommendation:** Model selection is fine. Tooling hygiene is the bottleneck.

OpenCode + qwen3:coder is sufficient.

**Action:** 
- Require tests alongside code
- Audit token usage per task
- Define maximum task scope explicitly

### 6. tasks.json vs. SQLite
**Recommendation:** Migrate to SQLite now.

Concurrent-write corruption is a ticking clock.

**Action:** Create TaskRepository abstraction wrapping SQLite. Persist to SQLite, optionally sync to JSON for backward compatibility.

## Additional Risks Identified

1. **Cron job drift/overlap:** Race conditions at interval boundaries. Implement job coordinator (SQLite lock table or flock).

2. **Dashboard split-brain:** Badge polling but page doesn't refresh = trust erosion. Add 30s silent refresh or downgrade badge.

3. **$10 OpenCode credit cap:** 16 days runway at 2 tasks/night. Single retry loop could exhaust. Set per-day spend limits and dead-man's switch.

4. **No per-job retry configuration:** Global 3-retry burns credits on guaranteed failures. Add retryable: boolean flag.

## Architectural Priorities (Impact/Effort)

1. Migrate tasks.json to SQLite (prevents data loss, enables querying)
2. Implement dedup keys for incidents (stops incident spam)
3. Add per-task retryable flag (stops credit waste)
4. Profile Agent Office performance (data before optimization)
5. Add main dashboard silent refresh (fixes split brain)

## What I'd Do Differently

1. Ship incident dedup BEFORE auto-resolution (reduce noise first)
2. Avoid "cron jobs" for real-time systems — use long-running process with timers
3. Separate Operations Engine into layers (retry, persistence, observability)
4. Add system health dashboard before more features
